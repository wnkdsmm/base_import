# IDEF0 Последовательная Архитектура Fire Data Pipeline v2.0

> **Цель документа:** Устранить возвратные связи из IDEF0_Единый_отчет и выстроить строгий последовательный поток A1→A2→A3→A4→A5→A6→A7→A8→A9, при котором каждый блок получает данные только от предшественника и передаёт их только преемнику.

---

## 1. Контекстная диаграмма A-0

### Система: Fire Data Pipeline — система поддержки принятия решений по пожарной безопасности

```
┌──────────────────────────────────────────────────────────────────────────┐
│  CONTROLS (управляющие воздействия)                                       │
│  • Нормативные требования по пожарной безопасности                        │
│  • Пороги качества данных (NULL_THRESHOLD, DOMINANT_VALUE_THRESHOLD)      │
│  • Регламенты хранения и обновления данных                                │
└────────────────────────┬─────────────────────────────────────────────────┘
                         │
```

## 2. Диаграмма декомпозиции A0 — Последовательный поток

### Принципиальное отличие от предыдущей версии

| Проблема в v1 | Решение в v2 |
|--------------|-------------|
| A4 вызывает A5 внутри себя (возвратная связь) | A5 получает результаты A4 как входные данные |
| A8 пронизывает все блоки A1–A9 (cross-cutting) | A8 — отдельный блок оркестрации между A7 и A9 |
| A7 строит собственную кластеризацию (DBSCAN) параллельно A3 | A7 потребляет кластеры A3 как входные данные |
| A3 и A4 выполняются независимо, без явного порядка | A3 → A4: кластеры используются для территориального прогноза |
| A9 запрашивает SQL сам (дублирует A1) | A9 потребляет только предварительно вычисленные payloads |

## 3. Детальное описание блоков

---

### A1: Загрузка и ETL данных о пожарах

**Назначение:** Принять сырые файлы от пользователя, провести валидацию, нормализацию и импорт в PostgreSQL, подготовить чистые таблицы для последующего анализа.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | Файлы пользователя (XLS/XLSX/CSV); параметры `project_name`, `output_folder` |
| **CONTROL** | Форматные правила (ImportDataStep); пороги качества `NULL_THRESHOLD=0.9`, `DOMINANT_VALUE_THRESHOLD=0.85`, `LOW_VARIANCE_THRESHOLD=0.0001`; реестр обязательных признаков (MANDATORY_FEATURE_REGISTRY) |
| **OUTPUT** | Сырая таблица в PostgreSQL (`project_name`); очищенная таблица (`clean_*`); профилировочный отчёт (*.csv, *.xlsx); лог импорта (JobStore) |
| **MECHANISM** | FastAPI (`/upload`, `/import_data`, `/run_profiling`); pandas; SQLAlchemy; NatashaColumnMatcher; ThreadPoolExecutor + JobStore |

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|
| A1.1 | Приём и валидация файла | Загруженный файл | Статус валидации, схема файла |
| A1.2 | Импорт в БД | Схема файла | Сырая таблица PostgreSQL |
| A1.3 | Очистка и нормализация | Сырая таблица | Нормализованная таблица (boolean coercion, pruning) |
| A1.4 | Профилирование признаков | Нормализованная таблица | Отчёт: null-доля, константность, дисперсия |
| A1.5 | Финализация и экспорт | Очищенная таблица | `clean_*` таблица, CSV/XLSX экспорт |

#### Ключевой алгоритм
`NatashaColumnMatcher` — гибридный матчер: mandatory → legacy → keyword → NLP fallback. Обеспечивает сопоставление заголовков столбцов с ожидаемыми признаками.

**→ Передаёт в A2:** `clean_*` таблицы + профилировочный отчёт

---

### A2: Машинное обучение и бэктестирование

**Назначение:** Обучить модели подсчёта/событий на временном ряду пожаров, провести бэктестирование, сформировать точечные прогнозы и доверительные интервалы.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | `clean_*` таблицы (дата, счётчик, тип, причина); температурные данные; параметры: горизонт прогноза, название таблицы, территория |
| **CONTROL** | Методы ML: LogisticRegression, PoissonRegressor, NegativeBinomial GLM; конформное предсказание (alpha=0.2); бэктест rolling-origin (MAE, RMSE, sMAPE, Poisson deviance); `ML_CACHE_SCHEMA_VERSION` |
| **OUTPUT** | ML-прогноз (N дней: point + lower/upper); важность признаков (permutation-based); метрики бэктеста; статус задачи (JobStore) |
| **MECHANISM** | scikit-learn (PoissonRegressor, LogisticRegression, StandardScaler); statsmodels (GLM NegBin); joblib; CopyingLruCache (128 items); ThreadPoolExecutor |

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|
| A2.1 | Загрузка и агрегация обучающей выборки | `clean_*` таблица | Дневные агрегаты пожаров |
| A2.2 | Инженерия признаков | Дневные агрегаты | Лаги, скользящие, температурные статистики |
| A2.3 | Обучение модели | Матрица признаков | Обученная модель (Poisson/NegBin/Logistic) |
| A2.4 | Бэктестирование | Модель + история | Метрики: MAE, RMSE, sMAPE, Poisson deviance |
| A2.5 | Калибровка интервалов | Бэктест residuals | Конформные границы (q-уровень) |

#### Ключевой алгоритм
Rolling-origin бэктестирование с 8–45 окнами. Выбор метода по минимальному Poisson deviance. Конформные интервалы через хронологическое разбиение.

**← Получает от A1:** `clean_*` таблицы  
**→ Передаёт в A3:** ML-прогноз + важность признаков (для контекста кластеризации)

---

### A3: Пространственная кластеризация территорий

**Назначение:** Выявить территориальные кластеры по пожарной активности, характеристикам ущерба и категориям пожаров.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | `clean_*` таблицы (агрегированные признаки по территориям: частота, площадь, ущерб, категории); ML-контекст от A2 (для приоритизации признаков) |
| **CONTROL** | Алгоритмы: KMeans / AgglomerativeClustering / Birch; диапазон k (CLUSTER_COUNT_OPTIONS); метрики качества: силуэт, Davies-Bouldin, Calinski-Harabasz |
| **OUTPUT** | Метки кластеров по территориям; профили кластеров (центроиды, средние); метрики качества; PCA-проекция (Plotly) |
| **MECHANISM** | scikit-learn (KMeans, Birch, PCA, StandardScaler, silhouette_score); pandas + numpy; Plotly; CopyingTtlCache (120s) |

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|
| A3.1 | Подготовка и нормализация данных | Агрегаты по территориям | Нормализованная матрица признаков |
| A3.2 | Выбор алгоритма и числа кластеров | Матрица признаков | Оптимальный k (по quality_score) |
| A3.3 | Выполнение кластеризации | Матрица + k | Метки кластеров для каждой территории |
| A3.4 | Оценка качества | Метки кластеров | Силуэт, DB, CH, баланс кластеров |
| A3.5 | Географическая интерпретация | Метки + координаты | Профили кластеров, представительные точки |

#### Ключевой алгоритм
Weighted KMeans с `incident_log`-весами. Автоматический выбор k через `quality_score = f(silhouette, DB, balance)`.

**← Получает от A1–A2:** `clean_*` таблицы + ML-контекст  
**→ Передаёт в A4:** Метки и профили кластеров для территориально-осознанного прогноза

---

### A4: Сценарный прогноз пожарной активности

**Назначение:** Построить сценарные прогнозы (baseline / оптимистический / пессимистический) с учётом сезонности, температуры и территориальных кластеров.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | `clean_*` таблицы (исторический ряд пожаров); профили кластеров от A3 (территориальный контекст); пользовательские фильтры: горизонт, таблица, история, причина/район |
| **CONTROL** | FORECASTING_FORECAST_DAY_OPTIONS; FORECASTING_HISTORY_WINDOWS; сценарные параметры; GEO_LOOKBACK_DAYS; MAX_GEO_HOTSPOTS |
| **OUTPUT** | Дневной прогноз (value, fire_probability, lower/upper bounds); объяснения прогноза (факторы, тренды); гео-горячие точки (высокоактивные зоны); рейтинг территорий по прогнозу |
| **MECHANISM** | app/services/forecasting (pipeline.py, assembly.py, sources.py); SQL-агрегации с кэшем TTL 120s; ThreadPoolExecutor |

> **Изменение v2:** A4 больше НЕ вызывает A5 внутри себя. Вместо этого A4 формирует `forecast_context` (территориальные прогнозы, горизонт), который передаётся в A5 как входные данные.

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|
| A4.1 | Выбор источника данных | `clean_*` таблицы | Исторические агрегаты |
| A4.2 | SQL-агрегация | Исторические данные | daily_history, option_catalog |
| A4.3 | Построение прогноза | History + кластеры A3 | Сезонный прогноз (weekday, month factors) |
| A4.4 | Доверительные интервалы | Прогноз + волатильность | Bounds = f(volatility × horizon) |
| A4.5 | Сборка прогнозного пакета | Все вычисления | forecast_payload + forecast_context для A5 |

#### Ключевой алгоритм
Коэффициенты сезонности (день недели, месяц) × коэффициент тренда × эффект температуры. Эвристические доверительные интервалы: `bounds = volatility × sqrt(horizon)`.

**← Получает от A3:** Профили кластеров  
**→ Передаёт в A5:** forecast_payload + forecast_context (территориальные прогнозы, горизонт)

---

### A5: Оценка территориального пожарного риска

**Назначение:** Ранжировать территории по интегральному риску, используя историческую статистику, прогноз активности от A4 и кластерный профиль от A3.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | `clean_*` таблицы (история пожаров по территориям); профили кластеров от A3; forecast_context от A4 (прогноз активности по районам, горизонт) |
| **CONTROL** | Весовые профили риска (adaptive / expert / calibratable); формула: Σ(weight_i × normalized_component_i); пороги надёжности (min объём данных); разрешение гео-сетки (GEO_LOOKBACK_DAYS) |
| **OUTPUT** | Рейтинг территорий (integral risk_score); компоненты риска (historical / forecast / structural); текстовые рекомендации; гео-горячие точки; флаги надёжности |
| **MECHANISM** | app/services/forecast_risk (core.py, scoring.py, geo.py, profiles.py, reliability.py); functools.lru_cache (32768); pandas + numpy |

> **Изменение v2:** A5 получает forecast_context от A4 как явный входной параметр — нет обратного вызова из A4.

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|
| A5.1 | Гео-привязка событий | Координаты пожаров | Ключи гео-ячеек `(floor(lat/size), floor(lon/size))` |
| A5.2 | Базовый расчёт риска | История + гео-ячейки | history_pressure, seasonal_alignment |
| A5.3 | Применение весового профиля | Базовый риск + профиль | Взвешенный интегральный балл |
| A5.4 | Ранжирование территорий | Интегральные баллы | Упорядоченный список по risk_score |
| A5.5 | Оценка надёжности | Объём данных + quality | confidence_score, флаги предупреждений |

#### Ключевой алгоритм
Grid-based geo-risk: `dominant_district` per cell; `recency_weight = max(0.2, 1 - age/GEO_LOOKBACK_DAYS)`; risk_score = Σ(component_weight × normalized_metric).

**← Получает от A3:** Профили кластеров  
**← Получает от A4:** forecast_context (территориальные прогнозы)  
**→ Передаёт в A6:** risk_context (рейтинг территорий для приоритизации анализа доступности)  
**→ Передаёт в A7:** geo_prediction, geo_summary, risk_zones

---

### A6: Анализ доступности пожарных подразделений

**Назначение:** Выявить объекты с неприемлемым временем прибытия пожарных расчётов, учитывая территориальный риск от A5.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | `clean_*` таблицы (объекты: название, адрес, координаты, категория, район); расстояние до ближайшей ПЧ (FIRE_STATION_DISTANCE); история инцидентов по объектам; risk_context от A5 |
| **CONTROL** | ACCESS_POINT_FEATURE_METADATA; OBJECT_CATEGORY_COLUMN_CANDIDATES; fallback-цепочка идентификации (адрес → объект → координаты → поселение → территория → район); LONG_RESPONSE_THRESHOLD=20 мин |
| **OUTPUT** | TOP-N проблемных точек доступа; компоненты балла (расстояние / частота / категория); текстовые объяснения; сводная статистика по зонам риска |
| **MECHANISM** | app/services/access_points (core.py, analysis.py, features.py, numeric.py); CopyingTtlCache (120s); pandas + numpy |

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|
| A6.1 | Извлечение и нормализация данных | `clean_*` + метаданные | Нормализованные признаки объектов |
| A6.2 | Разрешение географического ID | Данные объектов | Геопривязка через fallback-цепочку |
| A6.3 | Расчёт балла доступности | Нормализованные признаки | Балл = f(расстояние, время, вода, тяжесть, повторяемость) |
| A6.4 | Ранжирование и отчёт | Баллы + risk_context A5 | TOP-N объектов, пороги тяжести |

#### Ключевой алгоритм
Weighted score (94 балла) с штрафом неопределённости `δ = 6 × f(n_coverage)`. Fallback identity matching обеспечивает геопривязку даже при неполных данных.

**← Получает от A5:** risk_context (для приоритизации территорий)  
**→ Передаёт в A7:** access_points (проблемные объекты с баллами)

---

### A7: Интерактивная картографическая визуализация

**Назначение:** Отрисовать многослойную карту пожаров, интегрируя кластеры от A3, зоны риска от A5 и проблемные точки от A6.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | `clean_*` таблицы (координаты пожаров: lat, lon, date, атрибуты); кластеры от A3 (территориальные зоны); risk_zones + geo_summary от A5; access_points от A6 |
| **CONTROL** | core/mapping/config.py (параметры слоёв, стили); HTML-шаблоны (core/mapping/templates); параметры геометрии (analytics_geometry.py) |
| **OUTPUT** | `fires_map.html` (OpenLayers: точки, тепловая карта, кластерные зоны, зоны риска); `fires_map_analysis.json` (пространственная аналитика); `fires_map_analysis.md` (Markdown-отчёт); executive brief (приоритетные территории) |
| **MECHANISM** | MapCreator (mixin: Utilities, Analytics, Templates, Exports); analytics_hotspots.py; OpenLayers JS; OpenStreetMap; CopyingTtlCache (300s) |

> **Изменение v2:** A7 НЕ строит собственную кластеризацию (убран DBSCAN). Вместо этого использует кластеры, готовые от A3. Это устраняет дублирование и обеспечивает консистентность.

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|

### A8: Оркестрация и управление состоянием конвейера

**Назначение:** Принять все вычисленные payloads от A1–A7, закэшировать их с корректными TTL, обеспечить инвалидацию при изменении данных и сформировать единый пакет готовых данных для A9.

#### ICOM

| Категория | Элементы |
|-----------|---------|
| **INPUT** | Payloads от A1–A7 (profiling_report, ml_payload, cluster_payload, forecast_payload, risk_payload, access_payload, spatial_analytics) |
| **CONTROL** | Политики TTL: db_metadata=60s, dashboard_data=120s, dashboard_meta=300s, services=120s, fire_map=300s; LRU ML payload: max_size=128; политика freeze_mutable_payload; стратегия warmup |
| **OUTPUT** | In-memory кэши (TTL/LRU, thread-safe); быстрый доступ к метаданным (без повторных SQL); JobStore: статусы/логи/результаты задач; единый пакет данных для A9 |
| **MECHANISM** | app/cache.py (CopyingTtlCache/CopyingLruCache); app/runtime_invalidation.py; app/state.py (JobStore, SessionState); asyncio.create_task + asyncio.to_thread |

> **Изменение v2:** A8 перемещён между A7 и A9. Теперь это явный агрегирующий и кэширующий блок, а не невидимая cross-cutting инфраструктура. A8.3 (инвалидация) всё ещё связан с A1, но это CONTROL-связь (управление), а не возвратный DATA-поток.

#### Подфункции

| ID | Название | Входные данные | Выходные данные |
|----|---------|---------------|----------------|
| A8.1 | TTL-кэширование payloads | Результаты A1–A7 | Frozen payloads в памяти |
| A8.2 | LRU-кэширование ML | ml_payload от A2 | Быстрый доступ без повторного вычисления |
| A8.3 | Инвалидация при изменении данных | Сигнал из A1 (новый импорт) | Сброс устаревших кэшей A2–A7 |
| A8.4 | Управление состоянием задач | Статусы ThreadPoolExecutor | JobStore: статусы, логи, результаты |

#### Ключевой алгоритм
`CopyingTtlCache`: set сохраняет frozen payload ДО блокировки; get проверяет expiry и возвращает клонированную mutable копию. `freeze_mutable_payload`: рекурсивное преобразование dict→FrozenDict, list→FrozenList.

**← Получает от A1–A7:** Все вычисленные payloads  
**→ Передаёт в A9:** Пакет готовых данных (all_cached_payloads)

| **OUTPUT** | Метаданные дашборда (список таблиц, фильтры, доступные годы); payload (KPI-карточки, временная шкала, распределения, ущерб); Plotly-графики (JSON); управленческая сводка; таблица ущерба по типам объектов |

## 5. Устранённые возвратные связи

### 5.1 A4 → A5 → A4 (главная проблема v1)

**Было:** A4 вызывал `build_decision_support_payload()` внутри forecasting_pipeline, что инициировало полный цикл A5. Результат A5 встраивался обратно в A4.

**Стало:** A4 формирует `forecast_context` (территориальные прогнозы по дням/районам + горизонт) и передаёт его в A5 как входные данные. A5 выполняется ПОСЛЕ A4. A9 получает оба payload через A8.

**Что нужно изменить в коде:** В `app/services/forecasting/assembly.py` убрать вызов `build_decision_support_payload()` и вынести его в отдельный вызов после завершения forecast pipeline.

---

### 5.2 A8 → ALL (cross-cutting инфраструктура)

**Было:** A8 (кэш) был невидимой инфраструктурой, пронизывающей A1–A7. В IDEF0 это неправильно: кэш появлялся как MECHANISM у всех блоков, создавая скрытые зависимости.

**Стало:** A8 — явный блок с позицией в последовательности. A1–A7 всё ещё используют CopyingTtlCache как MECHANISM (это корректно — механизм, не поток данных). Но A8 как функциональный блок отвечает за агрегацию и финальное кэширование перед A9.

---

### 5.3 A7 → собственная кластеризация (дублирование A3)

**Было:** `analytics_dbscan.py` строил собственные зоны через DBSCAN независимо от A3.

**Стало:** A7 импортирует кластеры от A3. DBSCAN в A7 используется только для визуального разделения на карте (не для аналитических кластеров), что является деталью реализации визуализации, а не дублированием аналитики.
